## THE ROOT CAUSE (log-confirmed + audit-converged -- the win of NOT guessing)
The user's 1st hands-on: "Host takes piles -> client sees the pile still on ground" + "Client takes the pile host dropped -> pile converts into a ball, can be lifted, no sync." I READ THE LOGS (host + client `votv-coop.log`, 13:09) instead of guessing:
- The relay **WORKS** when eids match: the `5265 -> clump 5266 -> pile 5267` grab/hold/throw cycle completed cleanly on BOTH peers (logs prove it).
- The failures were piles at eids **45417 / 48074**. `kHostRangeSize = 32768` (element.h) -> those are **PEER-RANGE = the CLIENT minted them itself**. The host NEVER expressed them (empty express-grep). So the client was STILL AUTHORING its own duplicate shared-trash piles -- via a path PART 2 didn't cover: the **Init-POST spawn observer** (`prop_lifecycle.cpp`, the old "client interactions MUST propagate to host" fall-through for non-Aprop trash). The client-minted peer eid then **SHADOWS the adopted host eid in OnPileGrabPre** (GetPropElementIdForActor is checked BEFORE the mirror resolve) -> a re-grab relays a pileEid the host never had -> "no live pile" drop + a liftable local-only ball; and a host grab's PropDestroy(host eid) leaves the client's DUPLICATE copy on the ground.
- The **adversarial audit independently flagged the IDENTICAL bug as CRITICAL-1** -- two independent signals converged (the confidence the prior 3 rounds lacked).
- **FIX**: `prop_lifecycle.cpp:193` -- after the Aprop `IsDescendantOfProp -> return`, add `if (IsKeyedInteractable(self)) return;` for the client (a non-Aprop keyed-interactable IS shared trash). The client now authors NOTHING shared; the host's relay + grabs are the sole source. This COMPLETES the invisible-handle invariant: PART 2 (`EnsureHeldItemBroadcast`) gated the HELD-clump author path; this gates the SPAWNED-pile author path. RULE 2: retires the "client toClump/landing morph MUST propagate" path.

## LOAD-BEARING FACTS (do NOT re-derive)

- The client must author NOTHING shared-trash -- gated in TWO places now, BOTH `role==Client && non-Aprop keyed-interactable`: (a) `trash_collect_sync::EnsureHeldItemBroadcast` PART 2 (held clump), (b) `prop_lifecycle.cpp` Init-POST (spawned pile/clump). - The host's authoritative clump is a host-LOCAL (`MarkPropElement`, host-range) **pose-driven by the grabbing client** (incoming PropPose by the host clumpEid). This is the kerfur K-5 syncer model -- PROVEN to work (the host drives its own local kerfur prop from the client's stream). The clump spawn MUST silence the ground-hit self-convert (`SetActorRootNotifyRigidBodyCollision false`) or it spawns a 2nd pile.
- **VERIFY-DON'T-GUESS WINS** (the architect's design had 2 wrong assumptions the code-reading caught): (1) the eid-only convert fuzzy de-dupe is **DISABLED** for clumps (`remote_prop_spawn.cpp:580` `fuzzy = eidOnly ? nullptr`) -> the pending-adopt token is MANDATORY, not a fallback (a fresh-spawn would dupe). (2) the architect's single-packet throw (rely on the host clump's physics morph) was replaced with the **two-request model** (client death-watches its OWN BP clump -> relays PileThrowRequest) -- the BP morph is authentic + the host doesn't gamble on a bare-spawned clump's physics. (3) the client's OLD OnPileGrabPre PropDestroy was itself part of the bug (it wiped the host's pile with nothing in its place) -> replaced by the relay.
- `OnConvert` reuses `OnSpawn` for the newEid spawn (physFlags=0 settled) -> works for a clump newEid; disambiguates by oldEid. So GRAB = PropConvert(pile->clump), THROW = PropConvert(clump->pile), one adopt hook in OnConvert covers both.
- The eid=0 flood root was the literal "None" key passing the identity gate -- the fix is the same "None"->empty normalization the broadcast path already did.

## MTA PRECEDENT (cited; the model is MTA-faithful)
Server is the sole id-minter + broadcaster of create/destroy/mutate (`CStaticFunctionDefinitions::CreateObject/DestroyElement -> BroadcastOnlyJoined`); a client only fires a reliable REQUEST (`TriggerServerEvent -> Packet_LuaEvent`, gated `bAllowRemoteTrigger`); a client-created element is local-only `INVALID_ELEMENT_ID`, never broadcast; live pose is delegated to ONE syncer client (`CPedSync/CObjectSync`: server validates `GetSyncer()==player`, applies, relays to all others) while the server keeps identity. Our PileGrab/ThrowRequest = the request; the host's PropConvert = the broadcast; the grabbing client = the clump's syncer.
